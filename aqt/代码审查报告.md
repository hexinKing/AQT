## 一、安全

### [Critical] `config.py:11` — JWT 默认密钥暴露

**问题**：`jwt_secret` 默认值 `"change-me-in-production"` 是公开已知的字符串。若 `JWT_SECRET` 环境变量未设置，应用以此字符串签发 token，攻击者可伪造任意用户身份。

**修复**：删除默认值，启动时若未配置则抛出异常拒绝启动：
```python
@classmethod
def from_env(cls) -> "Settings":
    secret = os.getenv("JWT_SECRET")
    if not secret or secret == "change-me-in-production":
        raise RuntimeError("JWT_SECRET 未配置或使用了默认值，请在 .env 中设置随机字符串")
    return cls(jwt_secret=secret, ...)
```

---

### [High] `routers/settings.py:13-22` — SMTP 用户名随设置接口返回

**问题**：`GET /api/settings` 返回 `smtp_user`（邮箱地址），结合 XSS 漏洞可被读取。

**修复**：`SettingsOut` schema 移除 `smtp_user` 字段，前端从登录信息取用户名。

---

### [High] `schemas.py:25` — stock symbol 无格式校验

**问题**：`WatchlistCreate.symbol` 是裸 `str`，无长度或格式限制，任意字符串均可写入数据库并发送至 Tencent API。

**修复**：
```python
symbol: str = Field(..., pattern=r"^\d{6}$", min_length=6, max_length=6)
```

---

### [Medium] `config.py:13` — JWT 有效期 7 天无吊销机制

**问题**：token 泄露后攻击者有一周时间使用，无法提前失效。

**修复**：缩短至 24 小时，或在 `User` 表添加 `token_version` 字段实现服务端吊销。

---

### [Medium] `schemas.py:9` — 密码无长度限制

**问题**：用户可注册单字符密码，极易被暴力破解。

**修复**：`password: str = Field(..., min_length=8)`

---

### [Low] `models.py:24` — SMTP 密码明文存储

**问题**：`smtp_password` 以明文写入 SQLite，数据库泄露即等于 SMTP 授权码泄露。

**修复**：使用 `cryptography.fernet` 以服务端密钥对 SMTP 密码做可逆加密后存储。

## 二、正确性

### [Critical] `engine.py:92` — 日期比较使用本地时间而非北京时间

**问题**：`date.today()` 依赖服务器本地时区。若服务器时区早于 UTC+8（中国境外部署），每天开盘初数小时策略检查会被错误跳过。`_signal_exists_today`（第 24 行）和 `_count_signals_today`（第 39 行）同样以 UTC 午夜为"当天"边界，与北京时间差 8 小时。

**修复**：
```python
from datetime import timezone, timedelta
BJT = timezone(timedelta(hours=8))

def _today_bjt():
    return datetime.now(BJT).date()

def _today_start_bjt():
    return datetime.now(BJT).replace(hour=0, minute=0, second=0, microsecond=0)
```
在 `engine.py` 全部日期比较处替换为北京时间。

---

### [High] `strategies/trailing_stop.py:38` — 最高价 `highest_since_entry` 不持久化

**问题**：`params["highest_since_entry"] = highest` 在每次检查中更新，但 `engine.py` 仅在**产生信号时**将更新后的 `strategy_params` 写回数���库。若当天未触发止损，最高价更新丢失，下次运行时从旧值重新计算，导致止损线偏低。

**修复**：在 `engine.py` 策略循环中，无论是否产生信号，都将更新后的 `strategy_params` 写回数据库：
```python
# 在 signal = strategy.evaluate(...) 之后
item.strategy_params = new_params  # 始终持久化参数变更
db.commit()
```

---

### [High] `strategies/grid.py:33,44` — 网格容差带与相邻网格重叠

**问题**：固定 0.2% 容差在网格间距小时（如 `grid_pct=1%`）会导致相邻网格判定区重叠，价格穿越多个网格时仅触发最外层网格信号，中间层被跳过。

**修复**：记录上次触发的网格层级，仅在层级发生变化时产生信号；或动态计算容差为 `grid_pct * 0.1`。

---

### [Medium] `engine.py:88-95` — K 线数据不含今日时跳过整个 symbol

**问题**：盘前或 Tencent 数据延迟时，最新 K 线不是当天，导致整支股票被跳过，策略信号全部缺失。

**修复**：当最新 K 线非今日时，回退到使用实时报价作为当前价格，而非直接 `continue`。

---

### [Medium] `data_fetcher.py:309-310` — 分时图昨收价 fallback 不准确

**问题**：实时缓存为空时 `yesterday_close` 取 `prices[0]`（当天开盘价），导致分时图参考线位置错误。

**修复**：显式调用 `fetch_daily(symbol, days=2)` 取最近一个收盘日价格作为 `yesterday_close`。

---

### [Low] `risk.py:17-18` — 交易时段边界不精确

**问题**：下午收盘设为 14:55，实际 A 股 15:00 收盘，14:55-15:00 的信号被静默丢弃。

**修复**：`afternoon_end = time(15, 0)`，上午结束改为 `time(11, 30)`。

---

## 三、健壮性

### [High] `main.py:55` — 调度器关闭不等待任务完成

**问题**：`scheduler.shutdown(wait=False)` 在进行中的策略检查（包含数据库写入）未完成时即强制终止，可能导致数据不一致。

**修复**：`scheduler.shutdown(wait=True)` 并设置合理超时。

---

### [High] `notifier.py:27` — SMTP 异常被静默吞噬

**问题**：`except Exception: return False` 丢弃了具体错误，`_log_notification` 以空字符串记录错误原因，用户无法知晓邮件失败原因。

**修复**：
```python
except Exception as e:
    logger.exception("SMTP 发送失败")
    _log_notification(..., error=str(e))
    return False
```

---

### [High] `routers/dashboard.py:20` — `async def` 路由中调用同步 `requests`

**问题**：`kline` 和 `minute_kline` 端点声明为 `async def`，但内部调用 `fetch_daily`/`fetch_minute`（同步 `requests.get`），会阻塞事件循环，在并发请求时导致所有响应延迟。

**修复**：将这两个路由改为 `def`（FastAPI 会自动放到线程池执行），与其他路由保持一致。

---

### [Medium] `database.py:16-22` — WAL 模式写入无重试

**问题**：并发写入时仍可能遇到锁超时，策略引擎整批信号写入失败无重试，信号丢失。

**修复**：对写操作添加指数退避重试，或使用 `tenacity` 库的 `@retry` 装饰器。

---

### [Medium] `data_fetcher.py` — 网络请求无重试

**问题**：单次网络抖动导致整个检查周期数据缺失，且用户无任何提示。

**修复**：使用 `requests.adapters.HTTPAdapter(max_retries=Retry(total=3, backoff_factor=0.5))` 配置 Session。

---

## 四、代码质量

### [Medium] `data_fetcher.py:14-25` — 模块级可变缓存非线程安全

**问题**：`_daily_cache`、`_realtime_cache`、`_minute_cache` 被 FastAPI 请求线程和 APScheduler 后台线程并发读写，Python dict 的 check-and-set 操作不是原子的，可能发生竞态。

**修复**：使用 `threading.Lock` 保护写操作，或替换为 `cachetools.TTLCache`（线程安全）。

---

### [Low] `models.py:11` — `datetime.utcnow()` 返回 naive datetime

**问题**：naive datetime 与 aware datetime 混用时 Python 3 会抛出异常，且不携带时区信息，后续排查问题困难。

**修复**：使用 `datetime.now(timezone.utc)` 替代 `datetime.utcnow()`。

---

## 五、前端

### [High] `index.html:773` — `w.name` 未转义导致 XSS

**问题**：`` showKline('${w.symbol}','${w.name}') `` 直接将股票名称插入 HTML onclick 属性，名称含单引号时可注入任意 JS。

**修复**：
```js
const safeName = w.name.replace(/\\/g, '\\\\').replace(/'/g, "\\'");
// 或：用 dataset 传参，避免内联脚本
```

---

### [High] `index.html:765,815` — `JSON.parse(strategy_params)` 无异常处理

**问题**：`strategy_params` 格式异常时，整个看板渲染崩溃。

**修复**：
```js
function safeJsonParse(s, fallback = {}) {
    try { return JSON.parse(s || '{}'); } catch { return fallback; }
}
```

---

### [Medium] `index.html:412-417` — `showKline` 两个 API 串行调用，任意一个失败关闭弹窗

**问题**：日K成功但分时失败时弹窗被关闭，用户看到"加载失败"但实际日K数据已到位。

**修复**：使用 `Promise.allSettled` 并行请求，���时失败时仅隐藏分时标签，日K正常展示。

---

### [Medium] `index.html:686-703` — `localeCompare` 用作数值距离度量不正确

**问题**：`localeCompare` 返回 -1/0/1，不是时间距离，"找最近时间点"逻辑失效，分时图 x 轴标签映射错误。

**修复**：将时间字符串（`"HHMM"`）解析为整数分钟数后做数值差比较：
```js
const toMin = t => parseInt(t.slice(0,2))*60 + parseInt(t.slice(2,4));
```

---

### [Medium] `index.html:779` — sparkline 使用 `setTimeout` 延迟渲染，DOM 引用可能失效

**问题**：50ms 延迟后 Canvas 渲染与 DOM 更新之间存在竞态，动态渲染时 canvas 元素可能尚未挂载。

**修复**：移除 `setTimeout`，在 `innerHTML` 赋值后使用 `requestAnimationFrame` 或直接同步调用 `drawSparkline`。

---

### [Low] `index.html:282` — 页面头部显示邮箱而非用户名

**问题**：`j.data.email` 作为已登录用户名显示，未配置邮箱时显示空白。

**修复**：登录时将 `username` 存入 `localStorage`，优先使用用户名，邮箱作为备选。
