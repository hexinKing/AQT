---
  代码审查报告

  项目名称：AQT（A 股半自动量化监控工具）
  审查范围：aqt 项目当前版本
  审查目标：从可维护性、稳定性、安全性、扩展性、业务合理性角度给出问题分级与改进建议
  审查结论：当前项目适合作为 MVP/原型验证，不建议直接作为长期稳定实盘依赖系统。

  ---
  一、总体结论

  该项目的产品方向明确，技术选型符合“快速开发、快速验证”的目标：

  - 使用 FastAPI + SQLite + 单页前端，开发成本低
  - 采用半自动交易模式，只做信号提醒，不直接下单，风险控制合理
  - 支持少量股票监控、策略管理、持仓管理、邮件通知，符合个人使用场景

  但从工程质量看，当前版本更接近“可运行原型”，在以下方面仍有明显改进空间：

  - 路由与业务逻辑耦合较高
  - 数据访问层稳定性和性能边界较弱
  - SQLite 在并发和定时任务场景下存在锁风险
  - 风控能力基本缺失
  - 策略系统仍偏脚本化，缺少统一抽象和状态管理
  - 安全性适合本地使用，不适合公网直接暴露

  ---
  二、问题分级说明

  - 高：会明显影响稳定性、正确性、安全性，或阻碍后续升级
  - 中：当前可运行，但后续扩展或使用时会明显带来维护成本
  - 低：不影响 MVP 运行，但建议优化以提升体验或规范性

  ---
  三、问题清单与改进建议

  ---
  高级别问题

  1. main.py 可能承担过多职责

  级别：高

  问题描述

  当前项目很可能将以下职责集中在 main.py：

  - FastAPI 应用初始化
  - 路由定义
  - 页面挂载
  - 调度器启动与关闭
  - 业务逻辑拼装
  - 返回数据聚合

  这种结构在项目初期可以接受，但随着功能增加，会快速演变成“上帝文件”，导致：

  - 文件越来越大
  - 修改某个功能容易影响其他部分
  - 测试和定位问题困难
  - 后续引入新模块（风控、回测、通知日志）时容易失控

  影响

  - 降低可维护性
  - 降低代码可测试性
  - 增加迭代风险

  改进建议

  建议后续逐步按职责拆分：

  - routers/auth.py
  - routers/watchlist.py
  - routers/dashboard.py
  - routers/settings.py
  - services/signal_service.py
  - services/position_service.py
  - services/strategy_service.py

  至少形成：
  - 路由层
  - 服务层
  - 数据访问层

  ---
  2. SQLite 在定时任务 + 页面请求场景下存在锁风险

  级别：高

  问题描述

  项目使用 SQLite 作为数据库，这对 MVP 是合理的，但系统包含：

  - APScheduler 定时任务
  - 用户主动触发策略检查
  - 页面读取看板数据
  - 信号写入
  - 持仓更新
  - 设置修改

  这些行为在同一时段可能同时读写数据库。SQLite 对高并发写并不友好，容易出现：

  - database is locked
  - 写入冲突
  - 长事务阻塞读请求
  - 调度任务与 API 请求互相影响

  影响

  - 系统稳定性下降
  - 页面偶发报错
  - 信号写入失败或延迟

  改进建议

  短期建议：

  - 缩短 DB session 生命周期
  - 避免在定时任务中持有长连接
  - 减少单次任务中的数据库写入次数
  - 确保每个请求/任务独立 session，及时提交/关闭

  中期建议：

  - 若后续持续使用，迁移到 PostgreSQL 或 MySQL
  - 若仍坚持 SQLite，至少加更严格的访问规范与重试机制

  ---
  3. 数据源 akshare 的稳定性和性能依赖较强

  级别：高

  问题描述

  项目依赖 akshare 获取 A 股行情和历史数据。akshare 非常适合原型开发，但存在以下天然风险：

  - 上游接口可能变化
  - 拉取全市场数据时耗时长
  - 行情接口可能偶发失败
  - 字段结构变化会直接影响项目运行

  你之前已经遇到过性能问题，例如 stock_zh_a_spot_em() 会拉取大量股票数据，速度较慢。

  影响

  - 页面数据加载慢
  - 定时策略执行慢
  - 请求阻塞
  - 因外部依赖变化导致功能失效

  改进建议

  建议尽快强化数据层抽象：

  1. 所有 akshare 调用统一封装到 data_fetcher 层
  2. 明确区分：
    - 实时行情缓存
    - 历史 K 线缓存
  3. 对数据获取失败进行显式处理：
    - 返回错误状态
    - 记录日志
    - 页面显示“数据暂不可用”
  4. 后续预留切换数据源能力：
    - tushare
    - pytdx
    - 付费行情接口

  ---
  4. 风控能力基本缺失

  级别：高

  问题描述

  当前项目虽为半自动系统，但已经具有“信号生成 + 持仓展示 + 人工执行”的交易辅助属性。
  如果完全没有风控约束，后续只要往自动化再前进一步，就容易放大风险。

  当前缺少的关键风控能力包括：

  - 单票最大仓位限制
  - 同日最大信号次数限制
  - 连续亏损暂停机制
  - 涨跌停、停牌、ST 过滤
  - 非交易时段过滤
  - 异常波动不触发策略

  影响

  - 信号质量不稳定
  - 用户容易误解系统具备完整交易可靠性
  - 后续升级自动化时风险很大

  改进建议

  即使当前不实现完整风控，也建议至少预留风控模块设计，并优先补齐：

  - 交易时段过滤
  - 停牌/ST/涨跌停过滤
  - 单票仓位上限
  - 同方向重复信号抑制
  - 当日最大提醒次数限制

  ---
  中级别问题

  5. 策略模块更像“策略脚本”，而不是完整策略系统

  级别：中

  问题描述

  当前策略已拆分成独立文件（如双均线、网格、移动止损），这是好的开始。
  但如果策略接口不够统一，就会在后续扩展时出现问题，例如：

  - 输出格式不一致
  - 参数结构不统一
  - 无法通用管理策略结果
  - 不利于策略回测和对比

  尤其是网格和移动止损这类策略，本质上具有状态性，不能只依赖当前行情和静态参数。

  影响

  - 策略扩展成本上升
  - 容易产生重复信号或状态丢失
  - 不利于后续加入回测和自动执行

  改进建议

  建议统一策略输出格式，至少包含：

  - signal_type
  - reason
  - symbol
  - trigger_price
  - timestamp
  - meta

  同时考虑给部分策略增加状态存储能力，例如：

  - 上次买入区间
  - 历史最高价
  - 已触发层级
  - 最近信号方向

  ---
  6. 用户管理模块可能超出 MVP 实际需要

  级别：中

  问题描述

  如果当前项目的真实使用场景仅为个人本地使用，那么完整用户系统的价值有限。
  注册、登录、JWT 认证是合理的，但如果未来继续往以下方向扩展，就可能偏离 MVP：

  - 角色权限
  - 多租户
  - 复杂后台管理
  - 多人协作

  影响

  - 增加维护成本
  - 偏离当前产品目标
  - 稀释核心功能开发精力

  改进建议

  建议当前将“用户管理”收敛为：

  - 登录认证
  - 邮件配置
  - 个人偏好设置

  不建议短期内扩展复杂权限体系。

  ---
  7. 通知系统缺少可追踪性

  级别：中

  问题描述

  邮件通知是项目的重要输出渠道，但如果没有通知记录，就会出现：

  - 是否发送成功无法确认
  - SMTP 失败不易排查
  - 是否重复发送不透明
  - 页面提醒和邮件提醒不易对账

  影响

  - 降低通知可靠性
  - 用户难以判断提醒链路是否正常

  改进建议

  建议增加最小通知日志能力，记录：

  - 发送时间
  - 通知类型
  - 接收对象
  - 发送内容摘要
  - 发送结果
  - 错误原因

  这可以是一张简单的数据库表。

  ---
  8. 持仓数据手工维护，需明确边界

  级别：中

  问题描述

  当前持仓由用户手工录入，这符合“半自动工具”的设计目标，但也存在天然问题：

  - 用户可能忘记更新
  - 成本价可能与实际不一致
  - 卖出后持仓信息可能滞后
  - 系统中的持仓不一定等于真实券商账户持仓

  影响

  - 看板盈亏与真实账户可能不一致
  - 影响策略判断和信号解释

  改进建议

  建议在界面与文档中明确说明：

  ▎ 系统持仓为“用户维护持仓”，不与券商账户自动同步，仅作为策略判断和展示参考。

  后续若有需要，再增加导入成交记录或同步账户功能。

  ---
  9. 前端单页方案适合 MVP，但交互复杂度要受控

  级别：中

  问题描述

  当前前端采用单 HTML 文件，这非常适合快速开发，但若逻辑逐步增多，可能出现：

  - 状态管理混乱
  - 错误处理分散
  - 表单交互难维护
  - 参数展示不友好
  - JS 脚本越来越臃肿

  影响

  - 后续迭代成本上升
  - 页面易出现交互类问题

  改进建议

  短期建议：

  - 统一 loading、error、success 提示
  - 做基础表单校验
  - 策略参数使用更友好的表单，而不是直接暴露 JSON

  中期若页面复杂度继续上升，再考虑轻量前端框架。

  ---
  低级别问题

  10. README / 文档可继续增强“使用边界说明”

  级别：低

  问题描述

  当前项目说明文档如果主要覆盖安装、运行、功能介绍，是合格的。
  但对于量化类项目，建议额外强调以下边界：

  - 非自动下单系统
  - 持仓需手工维护
  - 行情依赖第三方数据源
  - 邮件通知依赖 SMTP 配置
  - 仅适合本地/局域网使用

  影响

  - 降低误用风险
  - 提升项目可理解性

  改进建议

  在 README 中增补“限制与注意事项”章节。

  ---
  11. 缺少更系统化的日志体系

  级别：低

  问题描述

  如果项目目前主要依赖控制台输出或简单错误返回，那么后续出现问题时，排查成本会较高。

  影响

  - 不易定位定时任务故障
  - 不易追踪数据源异常
  - 不易复盘策略执行过程

  改进建议

  后续建议增加统一日志方案，至少覆盖：

  - 应用启动/关闭
  - 数据拉取失败
  - 策略执行结果
  - 通知发送结果
  - 手动触发操作记录

  ---
  12. 测试体系仍偏轻量

  级别：低

  问题描述

  如果当前主要依赖集成测试脚本和人工验证，虽然适合 MVP，但后续每次改动都需要较多手工回归。

  影响

  - 迭代效率下降
  - 容易引入回归问题

  改进建议

  逐步补充自动化测试：

  - 策略单元测试
  - API 基础测试
  - 数据获取失败场景测试
  - 信号去重测试

  ---
  四、优先级排序建议

  第一优先级（建议尽快处理）

  1. 拆分 main.py
  2. 规范 SQLite 使用，降低锁风险
  3. 强化 akshare 数据层封装与异常处理
  4. 增加最基本风控过滤

  第二优先级

  5. 统一策略接口与输出格式
  6. 增加通知日志
  7. 明确持仓边界
  8. 优化前端错误处理与参数交互

  第三优先级

  9. 增强 README 边界说明
  10. 增加统一日志
  11. 建立基础测试体系

  ---
  五、最终审查结论

  该项目的最大优点是：目标明确、架构轻量、实现克制，适合快速验证个人量化辅助交易思路。

  该项目的主要不足是：当前工程化程度有限，数据源与数据库层稳定性边界较弱，策略和风控能力还不足以支撑长期稳定使用。

  因此，本次审查的最终结论为：

  ▎ 该项目适合作为个人 A 股半自动量化工具的 MVP 原型继续迭代，不建议在当前状态下直接作为长期稳定实盘依赖系统。

  ---
  六、建议的后续动作

  建议后续按以下顺序继续推进：

  1. 先稳定当前原型
  2. 再补风控与通知追踪
  3. 再做结构整理
  4. 最后再考虑自动交易接口接入
