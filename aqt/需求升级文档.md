# AQT 需求升级文档

> 创建日期：2026-07-14
> 优化日期：2026-07-14
> 基于版本：v1.0（scheduler 合并 + 代码审查后的版本）

---

## 1. 目标

在现有 AQT 能力（行情监控、策略信号、页面展示、邮件通知）基础上，新增三类能力：

- 信息面补充：自选股新闻聚合
- 被动复盘：每日收盘简报
- 辅助分析：AI 分析窗口

本次文档按“可直接开发”的粒度整理，重点明确：

- 功能范围
- 接口设计
- 数据结构
- 非功能约束
- 验收标准

---

## 2. 当前系统边界

现状约束如下：

- 后端：FastAPI + SQLite + SQLAlchemy + APScheduler
- 前端：单文件 `static/index.html`，无构建流程
- 看板聚合：后端统一返回 `positions`、`watchlist`、`signals`、`total_pnl`
- 邮件：已有 SMTP 发送能力，但当前为纯文本邮件
- 调度：当前基于工作日 + 时间窗，尚无交易日历能力
- 外部集成：当前没有新闻聚合、AI 模型调用、收盘快照持久化

实施时必须遵守：

1. 不改变现有策略引擎的核心行为
2. 新功能失败时不能拖垮 Dashboard、策略执行和通知主链路
3. 外部依赖必须有超时、降级和日志
4. 前端不可信，投资分析上下文由后端生成

---

## 3. 实施优先级

| 顺序 | 功能 | 阶段 | 原因 |
|------|------|------|------|
| 1 | 自选股新闻聚合 | Phase 1 | 价值直接、风险最低、最易验证 |
| 2 | 每日收盘简报 | Phase 1 | 与现有通知链路契合，适合先做结构化输出 |
| 3 | AI 分析窗口 | Phase 2 | 依赖外部模型、SSE、上下文治理，复杂度最高 |

结论：

- Phase 1 先交付“信息 + 复盘”
- Phase 2 再交付“AI 分析”

## 4. Phase 1：自选股新闻聚合

### 4.1 目标

让用户在 Dashboard 内看到与自选股和持仓相关的最新资讯，减少仅靠价格和信号判断的盲区。

### 4.2 范围

V1 只做：

- 基于自选股 symbol 的新闻聚合
- Dashboard 新闻卡片展示
- 基础缓存、去重、失败降级

V1 不做：

- 关键词订阅
- 板块新闻
- 新闻情绪分析
- AI 摘要

### 4.3 接口设计

`GET /api/news?symbols=600519,515050&limit=20&page=1`

请求参数：

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| `symbols` | `string` | 是 | 逗号分隔的股票代码 |
| `limit` | `int` | 否 | 返回条数，默认 `20`，最大 `50` |
| `page` | `int` | 否 | 分页参数，默认 `1` |

成功响应：

```json
{
  "ok": true,
  "data": {
    "items": [
      {
        "title": "白酒板块震荡走强",
        "source": "东方财富",
        "url": "https://example.com/news/123",
        "published_at": "2026-07-14 14:35:00",
        "symbols": ["600519"],
        "summary": "白酒龙头盘中走强，带动板块回暖。"
      }
    ],
    "total": 1,
    "page": 1,
    "limit": 20
  }
}
```

失败降级响应：

```json
{
  "ok": true,
  "data": {
    "items": [],
    "total": 0,
    "page": 1,
    "limit": 20
  },
  "message": "news source unavailable"
}
```

说明：

- 新闻源失败时仍返回 `200`
- 不让单个外部接口故障变成 Dashboard 整体故障

### 4.4 数据结构

后端内部建议定义新闻标准结构：

```python
{
  "title": str,
  "source": str,
  "url": str,
  "published_at": str,
  "symbols": list[str],
  "summary": str
}
```

字段要求：

- `title`：必填，最长建议 200 字符
- `source`：必填，来源名称
- `url`：必填，原文链接
- `published_at`：必填，统一格式 `YYYY-MM-DD HH:mm:ss`
- `symbols`：必填，至少包含一个 symbol
- `summary`：可空，若源站无摘要则返回空字符串

### 4.5 后端处理逻辑

处理流程：

1. 解析 `symbols`
2. 校验数量和格式
3. 对每个 `symbol` 拉取新闻
4. 聚合结果并标准化字段
5. 使用 `url + title + published_at` 去重
6. 按 `published_at` 倒序
7. 截断到 `limit`
8. 返回统一结构

强制约束：

- 单次请求最多处理 `20` 个 symbol
- 单个 symbol 抓取结果最多保留 `20` 条原始新闻
- 单个外部请求超时建议 `3-5s`
- 增加短 TTL 缓存，建议 `60-180s`
- 单个 symbol 拉取失败时，只记录日志，不中断其他 symbol

### 4.6 前端展示要求

Dashboard 新增“热点资讯”卡片：

- 默认显示最近 20 条
- 每条显示：标题、来源、发布时间
- 点击标题新窗口打开原文
- 加载中显示占位状态
- 无数据或失败时显示空状态文案

空状态文案建议：

- `暂无相关新闻`
- `新闻源暂不可用，请稍后再试`

### 4.7 验收标准

- 传入多个 symbol 时能返回合并后的新闻列表
- 重复新闻不会重复展示
- 新闻接口失败时 Dashboard 其他模块不受影响
- 接口在同一时间段内不会因频繁刷新反复命中外部源
- 前端能清晰区分“无新闻”和“新闻源失败”

## 5. Phase 1：每日收盘简报

### 5.1 目标

交易日收盘后，系统自动发送结构化复盘邮件，帮助用户快速回顾当天持仓和信号变化。

### 5.2 范围

V1 只做结构化简报，不依赖 AI。

V1 包含：

- 交易日定时发送
- 结构化邮件内容
- 失败日志记录
- HTML 邮件优先，纯文本为降级方案

V1 不做：

- AI 点评
- 图表内嵌邮件
- 日内开盘/昨收基准快照

### 5.3 调度要求

触发条件：

- 交易日执行
- 时间建议 `15:05` 或 `15:10`

调度规则：

- 不能只依赖周一到周五
- 节假日、休市日、调休日必须跳过
- 当日只允许发送一次

实现要求：

- 在调度模块新增独立 job
- 增加交易日判断函数
- 如行情不可用，可发送“部分数据缺失”的降级版本

### 5.4 邮件内容结构

邮件主题建议：

```text
[AQT 收盘简报] 2026-07-14
```

邮件内容建议包含：

1. 日期
2. 今日总浮盈/浮亏
3. 持仓摘要
4. 当日新增信号
5. 重点观察项

HTML 邮件示例结构：

```html
<h2>[AQT 收盘简报] 2026-07-14</h2>
<p>今日总浮盈：+¥1234.56</p>
<h3>持仓摘要</h3>
<table>...</table>
<h3>今日信号</h3>
<ul>...</ul>
<h3>重点观察</h3>
<ul>...</ul>
```

### 5.5 数据结构

建议后端内部生成统一的简报对象：

```python
{
  "date": "2026-07-14",
  "total_pnl": 1234.56,
  "positions": [
    {
      "symbol": "600519",
      "name": "贵州茅台",
      "shares": 100,
      "avg_cost": 1500.0,
      "last_price": 1512.3,
      "unrealized_pnl": 1230.0,
      "pnl_pct": 0.82
    }
  ],
  "signals": [
    {
      "symbol": "600519",
      "direction": "BUY",
      "strategy": "ma_cross",
      "reason": "MA5 上穿 MA20"
    }
  ],
  "highlights": [
    "600519 当日新产生 BUY 信号",
    "总持仓盈亏转正"
  ]
}
```

说明：

- `positions` 可复用现有 Dashboard 聚合后的扩展字段
- `signals` 只取当日新增信号
- `highlights` 先基于规则生成，不依赖 AI

### 5.6 数据来源与限制

V1 使用现有系统可稳定得到的数据：

- 持仓信息
- 当前行情
- 当日信号
- 当前总浮盈

V1 明确不承诺：

- 相对开盘盈亏
- 相对昨收盈亏
- 日内最大回撤

这些能力需后续补充快照机制后再支持。

### 5.7 失败与降级

- 邮件发送失败时记录通知日志
- 行情缺失时邮件仍可发送，但标记部分数据缺失
- 若用户未配置邮箱或 SMTP，不发送并记录原因
- 简报任务异常不能影响策略调度主任务

### 5.8 验收标准

- 交易日能在预定时间生成并发送简报
- 非交易日不会误发
- 邮件正文至少包含总盈亏、持仓摘要、当日信号
- SMTP 失败时有日志记录
- 行情部分缺失时仍能输出可读简报

## 6. Phase 2：AI 分析窗口

### 6.1 目标

提供一个基于当前账户状态的问答式分析入口，减少用户手动描述持仓、信号和盈亏状态的成本。

### 6.2 定位

AI 仅用于辅助分析，不参与自动交易，不覆盖原有策略结论。

页面需明确显示：

- `仅供参考，不构成投资建议`

### 6.3 范围

V2 包含：

- AI 聊天接口
- 右下角浮动面板
- 流式输出
- 快捷提问模板
- 本地历史会话

V2 不做：

- 自动交易建议执行
- AI 自动改写策略参数
- 长期历史会话服务端存储

### 6.4 接口设计

`POST /api/ai/chat`

请求体：

```json
{
  "messages": [
    { "role": "user", "content": "分析一下我的持仓风险" }
  ],
  "focus_symbol": "600519"
}
```

字段说明：

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| `messages` | `list` | 是 | 历史消息列表 |
| `focus_symbol` | `string` | 否 | 当前聚焦股票 |

响应方式：

- `text/event-stream`
- 增量返回模型输出

SSE 事件建议：

```text
event: chunk
data: {"content":"当前持仓中..."}

event: done
data: {}
```

错误事件建议：

```text
event: error
data: {"message":"ai service unavailable"}
```

### 6.5 上下文生成规则

后端负责组装 AI 上下文，不信任前端提交的持仓数据。

上下文建议包括：

- 当前持仓摘要
- 当前总浮盈/浮亏
- 自选股列表
- 最近 20 条信号
- 若传入 `focus_symbol`，则补充该标的相关行情和信号摘要

后端 system prompt 中必须加入：

- 角色：投资分析助手
- 输出约束：仅分析，不做收益承诺
- 风险提示：不构成投资建议

### 6.6 数据结构

请求消息结构：

```python
{
  "role": "user" | "assistant",
  "content": str
}
```

后端上下文对象建议：

```python
{
  "user_id": 1,
  "total_pnl": 1234.56,
  "positions": [...],
  "watchlist": [...],
  "signals": [...],
  "focus_symbol": "600519"
}
```

### 6.7 控制项

必须加入以下限制：

- `messages` 最多保留最近 `20` 条
- 单条消息长度限制，例如 `2000` 字符
- 总上下文长度上限
- 模型响应长度上限
- 月度调用成本控制
- 外部模型超时控制

### 6.8 前端交互要求

- 右下角浮动按钮打开侧边面板
- 面板包含 `新闻` / `AI 分析` 两个 Tab
- 预置快捷提问按钮
- 历史会话保存在 `localStorage`
- 发送中禁用重复提交
- 流式渲染输出
- 失败时可重试

### 6.9 失败与降级

- AI 服务失败时不影响 Dashboard 与策略模块
- SSE 中断时前端显示错误提示
- 若模型不可用，可允许面板打开但禁止发送并提示原因

### 6.10 验收标准

- AI 上下文由后端生成，不依赖前端传入的持仓明细
- 流式输出可正常结束，并有明确 `done` 事件
- 失败时前端能看到可理解的错误提示
- 历史会话刷新后仍保留
- AI 功能异常不影响其他 API

## 7. 需要新增或调整的数据与配置

### 7.1 配置项

建议新增环境变量：

```env
ANTHROPIC_API_KEY=
AI_MODEL_NAME=
AI_MONTHLY_BUDGET=
NEWS_CACHE_TTL=120
NEWS_HTTP_TIMEOUT=5
MARKET_CLOSE_REPORT_TIME=15:05
```

说明：

- AI 相关配置仅在 Phase 2 生效
- 新闻缓存和超时配置在 Phase 1 即生效

### 7.2 数据模型调整建议

V1 可不新增数据库表也能先落地，但建议预留以下扩展方向：

- `notification_logs`：继续复用，记录收盘简报发送结果
- 后续如要支持日内/昨收对比，可新增 `portfolio_snapshots` 表
- 如后续要支持 AI 使用统计，可新增 `ai_usage_logs` 表

当前阶段不强制新增表的原因：

- 先验证业务价值
- 控制改动范围

## 8. 模块拆分建议

为避免继续把逻辑堆积到现有单文件中，建议按职责拆分：

后端建议新增：

- `aqt/services/news_service.py`
- `aqt/services/report_service.py`
- `aqt/services/ai_service.py`
- `aqt/routers/news.py`
- `aqt/routers/ai.py`

前端建议至少拆分为：

- 新闻面板逻辑
- AI 面板逻辑
- Dashboard 主渲染逻辑

即使仍使用原生 JS，也应减少单文件内无边界增长。

## 9. 分阶段验收清单

### Phase 1 验收

- `GET /api/news` 可稳定返回统一格式结果
- 新闻失败时 Dashboard 不报错、不空白
- 收盘简报在交易日定时发送
- 简报邮件可读，至少包含总盈亏、持仓摘要、信号摘要
- 所有新增功能有基本日志输出

### Phase 2 验收

- `POST /api/ai/chat` 可稳定返回 SSE
- AI 上下文后端生成
- 会话历史可本地保留
- AI 故障不影响主业务
- 前端明确展示风险提示文案

## 10. 工时评估

| 项目 | 预估工时 |
|------|------|
| 新闻聚合接口 + 前端卡片 | 0.5 - 1 天 |
| 收盘简报任务 + 邮件模板 | 0.5 - 1 天 |
| AI 接口 + SSE + 前端面板 | 1 - 2 天 |
| 联调、异常处理、验收 | 0.5 - 1 天 |

总计建议：`2 - 4 天`

说明：

- 若只交付 MVP，时间可进一步压缩
- 若要求稳定、可维护、有清晰降级策略，不应按 8 小时排期

---

## 11. 最终结论

本次需求升级可以做，但必须按工程边界收敛后执行：

- 先做新闻聚合和结构化收盘简报，再做 AI 分析窗口
- 新闻、邮件、AI 都要有超时、降级、日志
- AI 上下文只由后端生成
- 收盘简报必须引入交易日判断
- 前端应控制复杂度，避免继续无边界堆积

按本版文档实施，开发可以直接进入接口、数据结构和验收驱动的执行阶段。
